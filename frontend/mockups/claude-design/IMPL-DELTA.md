## 1. API response shape — additions and changes

### 1.1 Per-tool result counts (THE big one — Gal called this out)

The new dashboard hero shows a `Scanned by` row with each tool's result count baked in:

```
Trivy 0.52 · 7 findings        Semgrep 1.70 · 3 findings        15 posture checks · 12 pass
```

The same `ToolPillBar` appears on:
- Surface 1 (report card hero) — all tools `done` with result counts
- Surface 2 (assessment progress) — tools in `pending | active | done` states, no counts yet
- Surface 3 (assessment-complete summary) — all tools `done`, no counts (just identity)
- Surface 4 (onboarding step 3) — all tools `pending`, no counts

Today IMPL-0003 only returns `scanner_versions: {trivy: "0.52.0", semgrep: "1.70.0"}` (just versions, no counts) and a parallel `tool_states[]` (only state, no version, no result).

**Proposed: collapse both into one canonical `tools[]` payload** that is a first-class assessment field, present on every endpoint that touches an assessment. Each tool entry is identity + state + optional result.

```jsonc
"tools": [
  { "id": "trivy",    "label": "Trivy 0.52",       "version": "0.52.0", "icon": "bug_report",
    "state": "done",  "result": { "kind": "findings_count", "value": 7,  "text": "7 findings" } },
  { "id": "semgrep",  "label": "Semgrep 1.70",     "version": "1.70.0", "icon": "code",
    "state": "done",  "result": { "kind": "findings_count", "value": 3,  "text": "3 findings" } },
  { "id": "posture",  "label": "15 posture checks", "version": null,    "icon": "rule",
    "state": "done",  "result": { "kind": "pass_count",     "value": 12, "text": "12 pass" } }
]
```

State values: `pending | active | done | skipped`.
Result is omitted on `pending` and `active` tools (counts don't exist yet).
The frontend renders `label` directly; the backend is the source of truth for tool identity + version.

This means:
- `/dashboard` returns `tools[]` with all states `done` + counts (current assessment)
- `/assessment/status/{id}` returns `tools[]` with live states + counts populated as each scanner finishes (matches what the engine already emits step-by-step)
- The standalone `scanner_versions` field becomes redundant. Keep it only if removing it would break a settled consumer; otherwise drop in v0.2.

Counts derive from filtered finding queries:
- Trivy: `count(finding) WHERE source_type = 'trivy' AND assessment_id = X`
- Semgrep: `count(finding) WHERE source_type = 'semgrep' AND assessment_id = X`
- Posture: `count(finding) WHERE type = 'posture' AND assessment_id = X AND state = 'pass'` (or just compute pass_count from the categories breakdown already in the response)

### 1.2 Posture check state — replace `passed: bool` with the four-state `state` field

PRD-0004 Story 3 settled the four-state pattern (`pass | fail | done | advisory`). The new design uses it directly: `PostureRowDone` shows a `Draft PR ↗` link to the GitHub PR that fixed the check. Today's IMPL-0003 schema only has `passed: bool` + `grade_impact`, which collapses pass and done.

**Proposed schema change** for each posture check item in `/dashboard`:

```jsonc
{
  "name": "code_owners_exists",
  "display_name": "Code owners file exists",
  "category": "code_integrity",
  "state": "done",                  // was: passed: bool
  "grade_impact": "counts",         // unchanged
  "fixable_by": "codeowners",       // unchanged
  "detail": "Generated by OpenSec on 2026-04-23",
  "pr_url": "https://github.com/galanko/opensec-demo/pull/14"  // NEW, only when state == "done"
}
```

State semantics:
- `pass` — check ran, passed, no agent involvement
- `fail` — check ran, failed, may be fixable
- `done` — check failed previously, an OpenSec agent generated a PR that's been merged (or is open and the user has merged it)
- `advisory` — informational, doesn't count toward grade

**Source of truth for `done`:** a posture finding (`type='posture'`) with `status='remediated'` or `status='closed'` AND `raw_payload.pull_request.url` set. The dashboard endpoint joins those.

**Why this matters:** the design relies on the difference between `pass` (was always fine) and `done` (we fixed it for you, here's the PR) as a brand moment. Don't collapse them.

### 1.3 Per-category progress rail

Surface 1 shows `CI supply chain · 1 of 2` with a thin progress rail. Today's `/dashboard` returns `posture.categories[].checks[]` and the frontend can derive progress, but precomputing keeps the consumer simple:

```jsonc
"posture": {
  "pass_count": 12,
  "total_count": 15,
  "advisory_count": 3,                            // NEW
  "categories": [
    {
      "name": "ci_supply_chain",
      "display_name": "CI supply chain",
      "progress": { "done": 1, "total": 2 },     // NEW (excludes advisory)
      "checks": [ ... ]
    }
  ]
}
```

`progress.done` counts `state='pass'` and `state='done'`. Advisory checks are not counted in either numerator or denominator (they don't affect the grade). `advisory_count` is a top-level convenience for the "12 of 15 checks pass · 3 advisory" subtitle on the posture card header.

### 1.4 Unmet-criteria labels for the report-card subtitle

Surface 1's `CompletionCard` reads "*2 criteria remaining: pin CI actions to SHA, add code owners.*" — needs the labels of the unmet criteria, not just the count.

**Proposed:** return the criteria as a labeled list, not a flat snapshot:

```jsonc
"criteria": [
  { "key": "security_md_present",       "label": "SECURITY.md present",         "met": true },
  { "key": "dependabot_configured",     "label": "Dependabot configured",       "met": true },
  { "key": "no_critical_vulns",         "label": "No critical vulns",           "met": true },
  { "key": "no_high_vulns",             "label": "No high vulns",               "met": false },
  { "key": "branch_protection_enabled", "label": "Branch protection enabled",   "met": true },
  { "key": "no_secrets_detected",       "label": "No committed secrets",        "met": true },
  { "key": "actions_pinned_to_sha",     "label": "CI actions pinned to SHA",    "met": false },
  { "key": "no_stale_collaborators",    "label": "No stale collaborators",      "met": true },
  { "key": "code_owners_exists",        "label": "Code owners file exists",     "met": false },
  { "key": "secret_scanning_enabled",   "label": "Secret scanning enabled",     "met": true }
]
```

Backend keeps the existing 10-bool `CriteriaSnapshot` JSON in the DB; the API layer wraps it with labels at response time. Labels live in one place (server), so changing copy doesn't require a frontend ship.

### 1.5 Vulnerabilities — split Semgrep code findings out of the count

Surface 1 vulns card subhead reads `3 are code issues from Semgrep · may need manual review`. Surface 3 vulns card has a footer `Trivy · Semgrep`. Need to surface code-finding count separately:

```jsonc
"vulnerabilities": {
  "total": 10,
  "by_severity": { "critical": 0, "high": 2, "medium": 5, "low": 3 },
  "by_source": {
    "dependency": 7,                         // type='dependency'
    "code": 3,                               // type='code' (Semgrep)
    "secret": 0                              // type='secret'  (Trivy secret-scan)
  },
  "tool_credits": ["Trivy", "Semgrep"]      // populated only when there's at least one finding from that scanner this run
}
```

Already trivially derivable from the unified `finding` table per ADR-0027. No new DB work — just a different shape on the dashboard endpoint.

### 1.6 Assessment timestamps

Surface 1 hero shows `Last assessed: 2 minutes ago`. Surface 2 shows `Elapsed: 01:24`. Surface 2 also shows a `Previous assessment` sub-card with `Grade B · 8 of 10 criteria · 3 days ago`. All client-derivable from timestamps already on the `assessment` row, but they need to be in the response payload:

```jsonc
"assessment": {
  "id": "asm_...",
  "started_at":   "2026-04-25T11:08:00Z",
  "completed_at": "2026-04-25T11:09:24Z"     // null while running
}
```

`/dashboard` returns the completed timestamps from the latest `complete` assessment.
`/assessment/status/{id}` returns `started_at` (and `completed_at` becomes non-null when the engine finishes).

### 1.7 Step `hint` field for pending steps

Surface 2 pending step example: `Checking repo posture` with right-aligned hint `15 checks`. Today's `steps[]` schema has `result_summary` (used after done), but no equivalent for pending. Add an optional `hint: string | null`. Populated on the engine side at step initialization for steps with a knowable scope ahead of time.

### 1.8 Editorial hero copy ("Nearly there.")

The hero headline ("Nearly there." / "Almost done." / "Off to a strong start." / etc.) is derived from `criteria_met / criteria_total` + count of fixable failing posture checks. **Architect's call:** generate it backend-side or client-side?

- **Client-side (recommended).** The mapping is purely deterministic and the copy is editorial — it should live next to the React component that renders it so design changes don't need a backend trip.
- **Backend-side.** Centralizes copy. Useful if we ever want to A/B test or localize, but neither is on the roadmap for v0.2.

### 1.9 First-time interstitial flag (`firstAssessmentSeen`)

Surface 3 shows only "after the first assessment after onboarding, OR after a re-assessment when the posture check count changed." This needs a per-assessment "have we shown the summary yet?" flag.

**Proposed:** add `summary_seen_at TIMESTAMP NULL` column to the `assessment` table + a `POST /api/assessment/{id}/mark-summary-seen` endpoint. Frontend logic: if `latest_assessment.summary_seen_at IS NULL`, route to Surface 3. The interstitial's "View your report card" CTA fires the POST and navigates.

This survives reloads (localStorage doesn't, and we're a single-user product so no multi-device sync to worry about).

### 1.10 Severity tile colors — design ships a new "code" severity chip kind

Surface 3 vulns card uses a `SeverityChip` with five `kind`s: `critical | high | medium | low | code`. The `code` kind uses `bg-primary-container/45 text-on-primary-container` — different from medium (tertiary) and low (neutral).

This is purely frontend. No API change. But the `SeverityChip` component (new) needs a `kind="code"` variant.

### 1.11 What stays as-is in the API

- `quick_wins[]` shape unchanged (already matches Surface 3's three-bullet list)
- `grade` and `criteria_total` unchanged
- `POST /findings/{finding_id}/fix` unchanged from IMPL-0003 Epic 4
- ADR-0027 / ADR-0028 internals unchanged

## 2. Frontend tokens, utilities, and global CSS

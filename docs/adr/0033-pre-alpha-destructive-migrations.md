# ADR-0033: Pre-alpha destructive migrations

**Date:** 2026-04-25
**Status:** Proposed
**PRD:** PRD-0003 v0.2 (and any other pre-alpha plan that touches schema)
**Plan:** IMPL-0003-p2 (assessment v2 — phase 2)
**Relates to:** ADR-0027 (unified findings model), ADR-0032 (rev. 2 dashboard payload)

## Context

IMPL-0003-p2 needs to land the unified-findings rebuild from ADR-0027 — one `finding` table with a typed `type` column (`dependency` | `secret` | `code` | `posture`), and the deprecation of the parallel `posture_check` table. The original ADR-0027 plan called for a migration that:

1. Pre-checks for duplicate `(source_type, source_id)` pairs and aborts with a clear error if any exist.
2. Runs inside `BEGIN`/`COMMIT` so a failure leaves the DB intact.
3. Migrates failing/advisory `posture_check` rows into `finding` as `type='posture'`.
4. Drops the `posture_check` table only after the migration succeeds.

That plan is correct for a product with users. It is over-engineered for OpenSec right now:

- **Single-user community edition.** The deployment shape is one Docker container, one SQLite file, one operator. No fleet to migrate.
- **No external alpha users yet.** PRD-0004 (v0.1 alpha) merged on `main` (commit `d83fc7c`, 2026-04-24); the first external installs have not happened. The repo's only data is Gal's local dogfooding state.
- **The CEO has explicitly authorized destructive migrations** for this phase (2026-04-25): "we can obsolete/delete/truncate anything we have in the db... right now its ok, these changes don't need to support backward compatibility."

We considered three ways to shape the migration:

| Option | Sketch | Verdict |
|---|---|---|
| A · Non-destructive migration with pre-check + row-by-row migration | Original ADR-0027 plan: pre-check duplicates, abort on conflict, migrate posture rows into `finding`, drop the old table | Rejected for phase 2. Adds ~150 lines of pre-check + migration SQL, two error paths to test, and a recovery branch the operator never hits because there's only one operator and they're running it for the first time |
| B · Destructive migration: drop, recreate, repopulate from scanners | DROP `finding`; DROP `posture_check`; CREATE the new unified `finding` table; the next assessment run repopulates everything | **Chosen** for phase 2 |
| C · Wipe the whole DB on first boot of the new build | A "fresh start" flag in config that reinitializes everything | Rejected. Heavier than needed — the assessment, completion, workspace, and audit-log tables all carry useful state we don't need to throw away. Surgically dropping `finding` + `posture_check` is enough |

Option B trades migration safety for shipping speed and code surface. The trade is sound *only* because there is no one whose data we are about to destroy. The license to be destructive expires at v0.1 alpha tag — once the first external user installs OpenSec, every schema change is non-destructive again.

## Decision

For pre-alpha schema migrations (until the v0.1 alpha tag is cut), destructive migrations are explicitly authorized when they materially simplify the implementation. The license has three guard rails:

### 1. Scope: only data the operator can regenerate cheaply

Destructive migrations may drop, truncate, or rebuild tables whose contents can be regenerated by re-running an assessment, re-cloning the repo, or re-ingesting an integration payload. Concretely, this includes:

- `finding` (regeneratable by re-running an assessment)
- `posture_check` (same)
- `ingest_job` (transient by design)
- `agent_run` and chat history rows tied to a workspace (regeneratable by re-running the workspace)

It explicitly **excludes**:

- `app_setting` (provider, model, API key — costs the user something to re-enter)
- `integration_config` and `credential` (credentials)
- `assessment` and `completion` audit rows (the historical record the user has earned)
- `repo_settings` (the connected repo + PAT)

If a destructive migration would touch one of the excluded tables, it is not authorized — the migration must preserve the data, the same as a post-alpha migration.

### 2. The migration documents what it destroys

The migration file's leading comment must list, in plain language, every table it drops or truncates and what the user will need to do to repopulate. The release notes for the build that ships the migration repeat that list.

### 3. The license expires at v0.1 alpha tag

Once `v0.1.0-alpha` is tagged and the first external install link is published, this ADR is **deprecated**. Any future schema change must be additive or non-destructive against existing data. A successor ADR will be needed to authorize a destructive migration after that point — and the bar will be high (e.g., a security incident that requires data wipe, with a documented user-facing migration path).

For IMPL-0003-p2 specifically: drop `finding`, drop `posture_check`, recreate `finding` with the unified schema (per ADR-0027). The next assessment run repopulates from scratch.

## Consequences

**Easier:**

- IMPL-0003-p2 phase 2 (unified findings) ships a single `CREATE TABLE` migration plus DAO updates, no row-migration logic, no duplicate pre-check, no recovery branch.
- The `to_findings.py` mappers can be tested against an empty DB; no fixtures of the old `posture_check` shape need to be maintained.
- One less failure mode at install time.

**Harder:**

- Anyone running OpenSec from a pre-IMPL-0003-p2 build (i.e., Gal locally) will lose their findings + posture_check rows on the next run. They will need to trigger a re-assessment to repopulate. Documented in the migration file and the release notes.
- A future maintainer reading the migration file needs to understand that "DROP TABLE finding" was a deliberate one-time concession, not a pattern. The leading comment makes that explicit.

**Known gaps:**

- This ADR does not authorize destructive migrations for the agent-run / sidebar history tables, even though phase 2 might benefit from a clean reset there. If a phase-2 implementation hits that need, a separate ADR amendment is required — keep the scope tight.
- The "regeneratable cheaply" criterion is judgment-call; this ADR does not list a formal threshold. Reasonable people may disagree on whether a particular table qualifies. When in doubt, treat the table as preserved (the post-alpha rule).

## Revisit

When `v0.1.0-alpha` is tagged. At that point this ADR's status flips to **Deprecated**, and a follow-up doc commit removes the destructive-migration license from the project's working assumptions. Any future destructive change goes through a new ADR with explicit user-impact analysis.

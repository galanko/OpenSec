# V1.1 — Autonomous Remediation Pipeline

**Status:** shipping in PR #49 (`fix/mvp-integration`).
**Proven E2E by:** autonomous PR [galanko/OpenSec#51](https://github.com/galanko/OpenSec/pull/51) — `jinja2 >=3.1.0 → >=3.1.6` fixing CVE-2025-27516, opened without human intervention.

V1.1 turns the V1.0 four-agent MVP into a five-agent, one-click remediation pipeline. A user clicks "Fix it" on a finding and the system clones, analyses, plans, edits, pushes, and opens a draft PR on its own — ~7 minutes end-to-end.

---

## What's new

### 1. Evidence-collector agent (5th pipeline step)

New agent between `exposure_analyzer` and `remediation_planner`. Runs in JSON-only mode and produces a structured evidence object:

- `affected_files` — list of `{path, line, context, file_type}`
- `dependency_chain` — "pkg_a depends on pkg_b depends on vulnerable_pkg"
- `dependency_type` — `direct` / `transitive`
- `current_version` — version likely present in the repo
- `fix_safety` — `safe_bump` / `breaking_change` / `needs_migration` / `code_fix`
- `fix_safety_reasoning` — why that classification
- `test_coverage` — `{relevant_tests, has_coverage, notes}`
- `recommended_approach` — concise fix strategy
- `impact_assessment` — what changes, what risks

This output is persisted in the sidebar `evidence` section and injected into the planner/executor prompts.

### 2. One-click "Fix it" pipeline

`POST /api/workspaces/{id}/pipeline/run-all` runs all remaining agents sequentially as a background task. Stops on 2 consecutive failures (`max_consecutive_failures=2`) to prevent runaway loops. Exposed in the UI as a single "Fix it" action chip.

### 3. Remediation executor with tool access

The executor now routes through `AgentTemplateEngine` (Jinja2 template at `backend/opensec/agents/templates/remediation_executor.md.j2`) and is the only tool-enabled agent. It receives:

- `GH_TOKEN` injected from the GitHub integration credentials
- `OPENSEC_REPO_URL` injected from the integration config
- Auto-approved tool tiers for `bash`, `edit`, and `external_directory`
- 10-minute execution timeout / 3-minute stall timeout

The template guides it through clone → edit → `uv lock` → commit → push → `gh pr create --draft`.

### 4. Finding status auto-advance (WP6)

Status moves forward as agents complete:

| Agent result | Status transition |
|---|---|
| `finding_enricher` completes | `new → triaged` |
| `remediation_planner` completes | `triaged → in_progress` |
| `remediation_executor` → `pr_created` | `in_progress → remediated` |
| `validation_checker` → `fixed` | `remediated → validated` |

Transitions are forward-only — a later re-run can't regress a finding.

### 5. Approval + result cards

- `PlanApprovalCard` — review/edit the remediation plan before the executor runs.
- `RemediationResultCard` — shows `pr_url`, `branch_name`, `changes_summary`, `test_results`, and `error_details` with a direct link to the PR.

## What's fixed

- **`ContextDocument.generate()` crash** (commit `7405fd9`) — now accepts and renders a `remediation` block with PR URL, branch, changes, and test results.
- **Executor forced into JSON-only mode** (commit `eceeabe`) — the executor was incorrectly in the no-tool list. It now uses the Jinja2 template with tool access and env-var injection.
- **Evidence collector re-suggestion loop** (commit `f430b88`) — earlier tool-capable evidence collector was timing out on repeated clones and being re-suggested indefinitely. Moved to JSON-only mode; pipeline `run-all` now caps consecutive failures at 2.
- **`external_directory` permission blocked executor** (commit `f430b88`) — executor needs to `git clone` outside the workspace cwd. Added `external_directory` to the tool-agent auto-approve list.
- **Ruff E501 on structured-output contract** (commit `04d2c60`) — wrapped the `affected_files` JSON line in `_STRUCTURED_OUTPUT_CONTRACTS` to unblock CI.

## Known issues / deferred work

Documented in `docs/known-issues/v1.1-e2e-findings.md` and tracked in Linear.

- **Permission ask-mode deferred.** All tool-agent actions (`bash`, `edit`, `external_directory`) currently auto-approve so the one-click pipeline doesn't stall. OpenCode-style ask UX is a follow-up — backend plumbing (`permission.asked` SSE events, grant/deny endpoints) already exists in `executor.py`; missing piece is the frontend approval cards + per-agent policy.
- **Evidence collector is speculative.** JSON-only mode produces plausible-but-unverified paths. Will re-enable read-only `bash`/`grep`/`cat` once ask-mode lands.
- **Workspace UI: evidence-collection card** — evidence-collector output is persisted in the sidebar but has no dedicated card. Tracked as a separate Linear ticket.

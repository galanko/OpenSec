# V1.2 Grade-A Dogfood — UI/UX gaps for Product & UX

Collected while driving OpenSec end-to-end against its own repo (April 2026):
clean Docker → onboarding → assessment → posture fixes → remediation agents →
PRs merged → Grade A reached. Every gap below is a thing the *user* felt, not
a code smell — they need product + UX to decide the shape before we build.

Scope: UI/UX only. Bugs with a clear technical root cause (B15, B16, B17) are
tracked separately in the bug log + PR history.

---

## UX-G01. No re-assessment button in the UI

**Symptom:** After onboarding, the dashboard grows stale. The only way to
re-run the assessment is `POST /api/assessment/start` from a terminal. A
user who merges a posture PR and wants to see their grade move has no path.

**Where it bites:** Dashboard (both empty and report-card states) and the
posture card right after a draft PR is merged.

**Open design questions:**
- Does re-assessment live as a "Run assessment now" button on the dashboard,
  or automatically trigger after the posture PR merges (via webhook)?
- Do we show the old grade until the new scan completes, or flip to the
  running state immediately?
- Where does the button sit without becoming a "big red destroy" button
  that users hit accidentally?

---

## UX-G02. Posture checks don't refresh after a PR is merged

**Symptom:** User generates a draft PR via the posture agent, merges it on
GitHub, comes back to OpenSec — the check is still marked red. Only a
manual re-assessment (UX-G01) updates it.

**Related to UX-G01** but distinct: even if re-assessment exists, the
system has no awareness that a PR tied to a check has landed.

**Open design questions:**
- Poll GitHub for PR state per open posture workspace, or wait for webhook?
- Should the posture row show "waiting for merge" state while the PR is
  open, flipping to green automatically once GitHub says `merged`?

---

## UX-G03. Failed agent runs bury the raw log

**Partly addressed in this PR** (B16 adds `agent_log_tail` to the posture
status). Still open:

- Remediation-executor failures (finding workspaces) don't expose the same
  tail — the full response only lives in
  `data/workspaces/<id>/history/agent-response.txt`.
- No way to copy or download the log from the UI; operator debugging
  means shelling into the container.

**Open design questions:**
- One "agent log" panel per workspace kind, or a global log drawer?
- Download-as-file action vs. inline disclosure?

---

## UX-G04. No cancel button for a running agent

**Symptom:** Posture and remediation agents have a 10-minute timeout. If an
LLM gets stuck in a thinking loop, the user can only wait. Restarting
Docker is the blunt-force workaround.

**Open design questions:**
- Cancel via terminating the OpenCode subprocess, or via a cooperative
  stop flag the agent checks between tool calls?
- Confirm dialog, or one-click with undo?

---

## UX-G05. "Generate draft PR" double-click footgun

**Symptom:** On posture rows, the button stays enabled while a prior run
is in flight. Nothing stops the user from starting two workspaces for the
same check. We saw this in dogfooding — ended with two competing PRs for
the same SECURITY.md.

**Quick fix candidates:** Disable + spinner on the button while any
`running`/`queued` workspace exists for that check; display "Agent
running" label instead of the CTA.

---

## UX-G06. Normalizer silently returns null for some findings (B17 spillover)

**Symptom:** The LLM normalizer intermittently fails to parse structured
output on chunks of 3+ findings. Those findings land with
`plain_description: null`. The findings list shows them with no plain-
English summary and no explanation.

**Open design questions:**
- Inline "regenerate description" action per row?
- Batch "retry normalization on findings without descriptions"?
- When should we show a loading skeleton vs. fall back to the raw
  `description`?

---

## UX-G07. Posture check detail is raw JSON

**Symptom:** Users without repo-admin vocabulary see field names like
`protected_branch`, `required_pr_reviews_count`, `commit_signature_protection`
inside the "Check detail (JSON)" expander. No explanation of what those
fields mean or how to read pass/fail.

**Open design questions:**
- Do we write plain-English micro-copy per field, or only show the fields
  that are failing with inline help?
- Link out to GitHub docs, or keep the explanation in-app?

---

## UX-G08. No global "what's happening right now" view

**Symptom:** Workspaces, posture fixes, and assessments all emit status
that's only visible if you're on the right page. If the user navigates
away, they can't tell an agent is still working — no notifications, no
global status bar.

**Open design questions:**
- Activity drawer on the right rail?
- Toast-style progress cards that follow you across pages?
- A dedicated "Runs" page analogous to GitHub Actions?

---

## UX-G09. Onboarding is one-shot; switching repos is hidden

**Symptom:** The wizard lets you connect one repo, then hands you to the
dashboard. Switching to a different repo requires editing the integration
row in Settings — no obvious "scan a different repo" CTA.

**Open design questions:**
- Multi-repo view, or repo switcher in the top bar?
- Does each repo get its own grade, or do we score at the org level?

---

## UX-G10. Model/provider preflight is missing

**Symptom:** If the user's OpenCode model name is misspelled or the API
key is revoked, posture and remediation agents run for 30-60 seconds
before failing with a cryptic "model not supported" error. No way to
validate the model + key at settings-save time.

**Related to:** v1.1 known-issue #2 (credential persistence) but distinct —
this one is about *testing* the model, not persisting it.

**Open design questions:**
- "Test model" button next to the model selector?
- Automatic probe on save, with a green check / red X?

---

## UX-G11. No link from a finding to "which workspace tried to fix it"

**Symptom:** After running `run-all` on a finding, the finding row in
the list doesn't link to the workspace that handled it. The user opens
the workspace separately to see what happened.

**Open design questions:**
- Tiny workspace chip on the finding row, or inline latest-run status?
- Does the row link to the latest workspace, or to the latest *successful*
  workspace?

---

## UX-G12. Completion ceremony fires on stale payloads

Flagged in a prior session but worth re-stating for UX review: the
celebration animation can fire after the assessment drifts back below
Grade A (e.g. a new critical finding lands between PR merge and next
assessment). The backend guards against stale `completion_id` but the
UX question remains: how do we handle "user reached A, then lost it"
without feeling punitive?

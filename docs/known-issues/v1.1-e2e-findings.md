# V1.1 E2E — Known Issues & Bug Log

Surfaced during the V1.1 end-to-end run (import → workspace → run-all → autonomous PR #51). Each item is mirrored into Linear as a `bug` in `Backlog` so it has a durable tracker home.

---

## 1. Fine-grained PAT scope friction

**Symptom:** Executor completes the edit + commit, but push fails silently. Sidebar shows `status: failed` with an `error_details` 403 message. No pre-flight hint that the PAT is read-only.

**Repro:** Configure the GitHub integration with a fine-grained PAT that has `Contents: read` (or no explicit Contents scope). Run the pipeline. Executor will reach the push step and 403.

**Root cause:** The integration `test` endpoint only validates auth, not write scope. GitHub's `x-accepted-github-permissions` response header lists required scopes but we never surface it.

**Fix direction:** Add a scope probe to the GitHub integration `test` handler: attempt a no-op `POST /git/refs` (intentionally malformed so it doesn't create anything) and read `x-accepted-github-permissions`. If `contents=write` is missing, fail the connection test with a clear message listing the required scopes.

---

## 2. OpenCode model / API key reset on container restart

**Symptom:** After `docker compose restart` the pipeline fails on the first agent run because no model is configured. Must re-`POST /api/settings/model` and `/api/settings/providers/anthropic/credentials` before running anything.

**Root cause:** OpenCode stores the active model and provider credentials in memory inside the singleton process.

**Fix direction:** Persist model selection and credentials in the existing `data/opensec.db` settings table. Re-apply on backend startup via the existing settings routes.

---

## 3. Credential key-name confusion

**Symptom:** First-time users send `{"key_name": "github_token", ...}` and get a 400 without guidance. The integration expects `github_personal_access_token`.

**Fix direction:** On 400 from `/api/settings/integrations/{id}/credentials`, return the expected `key_name` in the error body (e.g., `{"expected_key_name": "github_personal_access_token"}`). Also surface the expected name in the integration registry metadata so the frontend can pre-fill it.

---

## 4. Evidence collector is speculative

**Symptom:** Evidence-collector output contains phrases like "Expected to contain…" and paths that weren't actually verified against the repo. The content is usually correct but it's not grounded in real file reads.

**Root cause:** When the tool-capable version of evidence_collector was still enabled, it would re-clone the repo, run `grep` loops, and time out at 11+ min without producing JSON — blocking the pipeline. We moved it to JSON-only mode in `f430b88` to keep the pipeline reliable.

**Fix direction:** Once the permission ask-mode UI lands (see `memory/project_permission_ask_mode.md`), re-enable a restricted tool set for evidence_collector — read-only `bash` (`grep`, `cat`, `ls`) with a hard 2-minute cap and explicit stall detection. Keep JSON-only as the fallback on cap.

---

## 5. Pipeline re-suggestion loop

**Symptom:** Before `f430b88`, an agent that completed but didn't update its sidebar section was re-suggested forever by `suggest_next`, and the pipeline never terminated.

**Current mitigation:** `max_consecutive_failures=2` in `run-all` stops runaway loops triggered by agent exceptions.

**Fix direction:** Also fail the run if `suggest_next` returns the same `agent_type` twice in succession even when both runs "succeeded" — this catches the case where the agent returned no usable output and the sidebar wasn't advanced.
